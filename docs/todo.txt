LocalDrop TODO

1. Transfer progress UX and logic
Status: confirmed issue, possible

Problem
- Sending works, but the progress presentation is still too coarse.
- Current UI shows one active `ActiveTransferProgress` sheet at a time, centered around the current file, not a full batch view.
- This creates the impression that file progress is inconsistent when sending multiple files.

Current code logic
- Runtime contract only emits a single stream item at a time via `TransferRuntime.progressEvents()`.
- `LocalSendRuntimeAdapter` builds batch progress in sequence inside `sendStagedItems(to:pin:)`.
- The adapter already calculates both aggregate and per-file values:
  - `totalBytes`
  - `transferredBytes`
  - `currentFileTotalBytes`
  - `currentFileTransferredBytes`
- `TransferFeatureStore` stores only one `activeTransfer`.
- `TransferProgressSheet` renders one progress bar from `progress.progress`.

Affected areas
- `Modules/FeatureTransfer/Sources/FeatureTransfer/Infrastructure/TransferRuntime.swift`
- `Modules/FeatureTransfer/Sources/FeatureTransfer/Infrastructure/LocalSendRuntimeAdapter.swift`
- `Modules/FeatureTransfer/Sources/FeatureTransfer/Models/FeatureTransferModels.swift`
- `Modules/FeatureTransfer/Sources/FeatureTransfer/Application/TransferFeatureStore.swift`
- `Modules/FeatureTransfer/Sources/FeatureTransfer/Sheets/TransferProgressSheet.swift`
- `Modules/FeatureTransfer/Sources/FeatureTransfer/MenuBarExtraView.swift`

What is possible now
- Show 2 progress sections in the existing sheet:
  - overall batch progress from `transferredBytes / totalBytes`
  - current file progress from `currentFileTransferredBytes / currentFileTotalBytes`
- Keep one active sheet, but make it visually match LocalDrop better.
- Show "file X of Y" and remaining item count during a batch.

What is not possible cleanly without model changes
- A true LocalDrop-style per-file progress list for the whole batch.
- Persisting completed progress rows while the next file continues.
- Independent per-file rows in menu bar / history during a live batch.

Needed if we want full parity
- Extend `ActiveTransferProgress` or introduce a batch progress model with:
  - total file count
  - current file index
  - per-file progress snapshots
  - completed file rows
- Update store and UI to render batch state instead of a single active item.

Recommended next step
- Implement dual-progress UI first.
- If that still feels insufficient, follow with a batch progress model refactor.

Implementation plan
- Keep the existing single active-transfer flow in place for this pass:
  - `TransferRuntime.progressEvents()` stays as a single-event stream.
  - `TransferFeatureStore.activeTransfer` stays a single value.
  - No attempt to build a persistent per-file batch timeline yet.
- Extend `ActiveTransferProgress` just enough to support batch-aware copy:
  - add `totalItemCount`
  - add `currentItemIndex`
  - add derived helpers for `overallProgress`, `currentFileProgress`, and safe batch labels
  - keep the type backward-compatible for receive flows by defaulting to `1 of 1`
- Update `LocalSendRuntimeAdapter.sendStagedItems(to:pin:)` to emit richer batch metadata on every send progress event:
  - populate `totalItemCount` from the accepted staged item count
  - populate `currentItemIndex` from the current send loop index
  - keep `totalBytes`, `transferredBytes`, `currentFileTotalBytes`, and `currentFileTransferredBytes` authoritative
  - preserve current failure/cancel/completion behavior and terminal events
- Update receive-side progress emission conservatively:
  - keep receive behavior as a single-file presentation unless the runtime already knows the batch position
  - emit `totalItemCount = 1` and `currentItemIndex = 1` when no better receive metadata exists
- Refactor `TransferProgressSheet` into a dual-progress sheet:
  - section 1: overall batch progress bar and transferred/total label
  - section 2: current file progress bar and current file byte label
  - show `file X of Y`
  - show remaining item count when `Y > X`
  - keep the existing completion burst, cancel action, and completion state behavior
- Update `MenuBarExtraView` copy only where it helps:
  - keep one active transfer title
  - prefer showing batch-aware wording when `totalItemCount > 1`, but do not introduce per-file menu rows
- Test the feature at the store/runtime seam instead of relying on manual UI inspection only:
  - add `FeatureTransferTests` coverage for batch progress updates carrying `currentItemIndex` and `totalItemCount`
  - add coverage for terminal completion preserving the final batch metadata long enough for the success state
  - add coverage for cancel/failure/reset still clearing `activeTransfer` correctly
  - add coverage for menu summary text when a multi-file batch is active
- Verification target for this pass:
  - `swift test --package-path Modules/FeatureTransfer`
  - `swift test --package-path Modules/LocalSendKit`
  - if progress-related seams changed outside the package boundary, finish with `xcodebuild -project LocalDrop.xcodeproj -scheme LocalDrop build`

Acceptance criteria
- Sending multiple files shows both overall batch progress and current-file progress in the sheet.
- The sheet clearly indicates batch position with `file X of Y`.
- The menu bar still shows only one active transfer, but its wording no longer implies that the batch itself is only one file.
- Cancel, failure, reset, and completed states behave exactly as they do now aside from the richer progress presentation.
- No full batch-history or per-file live-row model is introduced in this pass.
